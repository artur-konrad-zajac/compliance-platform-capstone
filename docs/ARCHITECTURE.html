Compliance Platform: ADK Architecture & AI Orchestration

1. High-Level Design (HLD)

The Compliance Platform is a distributed, AI-driven application built on Google Cloud Platform (GCP). It leverages the Agent Development Kit (ADK) to orchestrate a suite of specialized Large Language Model (LLM) agents powered by Gemini 2.5 Pro. The system's primary objective is to automate the translation of dense regulatory text into actionable compliance forms and to rigorously evaluate applicant responses against those regulations.

  • Cloud Run: Hosts the synchronous HTTP API and the asynchronous background workers. The background workers use a non-blocking queue processor with a concurrency cap to process multiple jobs simultaneously.
  • Firestore: Replaces traditional queueing (e.g., Cloud Tasks), acting as a highly resilient polling queue that manages ADK sessions.
  • Cloud Storage: Persists schemas, regulatory text, dynamic evaluators, and tenant data. Serves as a persistent cache for intelligent checkpointing and resuming of paused sessions.
  • Vertex AI (Gemini 2.5 Pro): Drives the reasoning engine for the agent swarms.
  • Cloud Logging (ADK Telemetry): The google_adk framework exports OpenTelemetry data (e.g., gen_ai.choice custom logs). Crucial: The backend's Cloud Run Service Account must have the roles/logging.logWriter IAM permission, otherwise the background unhandled exception will crash the entire agent loop.

1.2 Data Ingestion & Caching Strategy

The platform relies on processing massive regulatory documents dynamically fetched from external sources. To ensure high availability and ethical crawling:

  • EurLexFetcher (Cellar API Integration): The API uses the official EUR-Lex Cellar REST API (https://publications.europa.eu/resource/celex/) to reliably download raw regulatory text. This programmatic access method natively avoids AWS WAF bot-protection challenges present on the public HTML UI.
  • Cloud Storage Caching Layer: To drastically reduce Vertex AI context ingestion costs, the system implements a permanent caching mechanism in GCS.
  • Raw HTML/Markdown regulations are cached under cache/regulations/{celex_id}.md.
  • Finalized JSON form schemas are cached under cache/forms/{celex_id}.json.
  • Dynamically generated Evaluator Swarms are cached under cache/evaluators/{celex_id}.json.
  • When a user requests a form for a previously processed CELEX ID, the system bypasses the ADK swarm entirely and serves the cached artifact within milliseconds.

1.3 Dynamic Configuration Management (Firestore)

The backend actively manages its configuration state via the config collection in Firestore, ensuring that missing configurations are automatically backfilled on startup to maintain system integrity:

  • agent_params: Controls the core behavior of the AI swarm (e.g., legal_remark_weight, ux_remark_weight, max_iterations, max_defects_per_iteration). If any parameters are missing, the backend auto-injects hardcoded defaults (e.g., weights default to 1, max iterations to 30) using a smart merge=True operation.
  • limits: Enforces strict tenant and global limits (e.g., max_daily_sessions, max_document_size_mb). Defaulting strictly limits resource consumption out-of-the-box (e.g., max 20 daily sessions).
  • rate_limits: Manages all AI API interaction quotas to prevent run-away AI execution and unbounded vertex AI billing (e.g., global_api_max_requests, ask_ai_window_seconds).

2. Low-Level Design: Agent Orchestration Patterns

2.1 Responsibility Separation Prompting (RSP)

Instead of relying on a single monolithic prompt, the cognitive load is distributed across a Sequential Pipeline (Distillation -> Architecture -> Building -> Validation).

2.2 Pre-Emptive "Reviewer 2" Defenses (Adversarial AI Security)

To ensure system safety against prompt injection and hallucination, the architecture heavily employs a Maker-Checker workflow. Adversarial agents ("Reviewer 2") rigorously audit inputs and outputs: - document_security_guard: Audits Applicant-uploaded PDFs to block hidden prompt injections before the auto-fill agent reads them. - prompt_security_guard: Audits the dynamically generated JSON Evaluators to ensure they don't contain malicious meta-instructions. - evaluation_audit_agent: Audits the final Compliance Report generated by the Swarm Aggregator, overriding the grade if the Aggregator was tricked into a false approval.

2.3 Dynamic Fan-Out / Fan-In Swarm Aggregation

Instead of hardcoded domain evaluators, the system uses a dynamic_creator_agent to read the regulation and instantly generate a customized array of Evaluation Modules (Rulesets). The dynamic_aggregator then compiles findings by applying these rulesets to evaluate the form in a single report, enforcing Priority Stacking rules (Accuracy > UX). By generating Evaluation Modules rather than autonomous agent prompts, we prevent "prompt entropy" and collisions within the Swarm Aggregator's context window.

2.4 State 4 Self-Improvement

A meta-agent analyzes session logs to automatically propose system-wide prompt mutations to prevent hallucination entropy over time.

Debate Loop Constraints & Forward-Rolling Patches: To prevent "prompt entropy" and infinite loops during the Consensus phase, the UX and Legal Critique agents output a specific {"defect_count": N} metric capped by a configured limit. The system strictly rolls forward, avoiding explicit rollbacks that were prone to false positives when scores oscillated due to defect batching. Instead, it relies on the Patch Verifier to ensure syntactic validity (retrying internally without rolling back) and a Historical Decisions ledger to provide contextual memory so agents know what they've previously attempted. Furthermore, the schema architecture physically decouples entities (dictionaries) from layout (arrays) to ensure deterministic patching.

21. Dynamic Fan-Out / Fan-In Swarm Aggregation (dynamic_aggregator)

Compiles findings from the dynamically loaded GCS Evaluators into a unified JSON Assessment Report using Priority Stacking.

System Prompt (Instruction)

[System Persona]
You are the Swarm Aggregator Agent, an objective legal auditor.

[Task]
Evaluate the form against {target_reg} and compile findings from the following specialized agents into a unified, strictly cited report:

Specialized Evaluators:
{agents_context}

[Rules]
1. Priority Stack: Legal Accuracy > Completeness of JSON Schema > UX Brevity. If these conflict, ALWAYS prioritize Legal Accuracy.
2. Use objective, deterministic legal language. Focus on verifiable facts and actions.
3. You must process and include an entry in `question_actions` for EVERY SINGLE QUESTION present in the form configuration. Do not skip any question, regardless of its type.
4. IMPORTANT: NEVER use internal question IDs (e.g. q_b0102_list_entities_in_scope) in the text of your `explanation` or `customer_comment`. Instead, always refer to the human-readable question text or label provided in the schema.

[Evaluation Logic]
- **Restricted Inputs (radio, dropdown, checkbox, toggle):** If the applicant selects a legally compliant option, output `recommendation: Approve` without requiring further context. If they select a non-compliant option, output `recommendation: Reject`.
- **Free-Text & File Uploads:** Evaluate if the content provides specific, actionable details relevant to the regulation, or if the file clearly relates to the requested evidence.
- **Failure Path (Free-Text/File):** If the text is a single word (e.g., "Yes") or lacks specific context, OR if a file upload is clearly a placeholder or irrelevant, output `recommendation: Reject` and set `customer_comment` to request the specific missing details or correct documentation.

[Few-Shot Examples]
*Tone and Language:*
- Bad Evaluation: "We feel that this answer is insufficient."
- Good Evaluation: "We determine that this answer lacks the required detail under Article 15."

[Output Schema]
You MUST output your response strictly as a JSON object exactly matching this schema:
{{
  "recommendation": "approve|reject|needs_revision",
  "general_comment": "A professional and polite general feedback comment regarding the form directed to the applicant. OMIT if not needed.",
  "question_actions": [
    {{
      "id": "question_id_here",
      "recommendation": "Approve|Reject",
      "explanation": "Detailed explanation for the internal reviewer of WHY you are approving/rejecting.",
      "customer_comment": "A professional and polite question or request directed to the applicant. OMIT or leave empty if there is no need to ask the applicant anything."
    }}
  ]
}}
